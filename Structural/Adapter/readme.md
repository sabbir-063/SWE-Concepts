# 6. Adapter Pattern (অ্যাডাপ্টার প্যাটার্ন)

### 📖 মূল কনসেপ্ট (Definition)
**সহজ বাংলায়:** আপনার কাছে দুটি আলাদা জিনিস আছে, যারা একে অপরের সাথে সরাসরি কথা বলতে বা কাজ করতে পারে না কারণ তাদের ইন্টারফেস বা প্লাগ আলাদা। অ্যাডাপ্টারের কাজ হলো এই দুজনের মাঝে একটি ব্রিজ বা অনুবাদক হিসেবে কাজ করে তাদের জুড়ে দেওয়া। 
যেমন: ৩-পিনের প্লাগকে ২-পিনের সকেটে ঢোকানোর জন্য আমরা যে মাল্টিপ্লাগ বা অ্যাডাপ্টার ব্যবহার করি, এটি ঠিক তাই!

**In English:** Adapter pattern allows objects with incompatible interfaces to collaborate.

---

### 💻 কিভাবে কাজ করে? (Implementation Details)
1. ক্লায়েন্ট একটি নির্দিষ্ট ইন্টারফেস (Target Interface) ব্যবহার করে কাজ করতে অভ্যস্ত।
2. কিন্তু আপনার কাছে যে সার্ভিস ক্লাসটি (Adaptee) আছে, তার ইন্টারফেস আলাদা।
3. আপনি একটি Adapter ক্লাস বানাবেন যা Target Interface ইমপ্লিমেন্ট করবে, এবং ভেতরে Adaptee-এর একটি অবজেক্ট ধারণ (Composition) করবে। 
4. ক্লায়েন্ট যখন অ্যাডাপ্টারকে কল করবে, অ্যাডাপ্টার সেই কলটিকে অনুবাদ করে Adaptee-কে বুঝিয়ে দিবে।

---

### 🧪 বাস্তব জীবনের উদাহরণ (Industry Use Cases)
1. **Legacy Code Integration:** পুরনো সিস্টেমের (Legacy) কোডের সাথে নতুন সিস্টেমকে জুড়ে দেওয়ার জন্য এটি প্রচুর ব্যবহৃত হয়।
2. **Third-party APIs:** থার্ড-পার্টি লাইব্রেরির ডাটা ফরম্যাট (যেমন XML) যদি আপনার অ্যাপের ফরম্যাটের (যেমন JSON) সাথে না মেলে, তবে আপনি একটি অ্যাডাপ্টার ক্লাস দিয়ে তা কনভার্ট করতে পারেন।

---

### ✅ সুবিধা (Pros)
- **Single Responsibility Principle:** ডাটা কনভার্সন বা অনুবাদের কাজটি মূল কোড থেকে আলাদা হয়ে যায়।
- **Open/Closed Principle:** নতুন ধরনের অ্যাডাপ্টার যুক্ত করলে ক্লায়েন্টের কোড ভাঙে না।

### ❌ অসুবিধা (Cons)
- কোডের কমপ্লেক্সিটি বাড়ে, কারণ অনেক নতুন ক্লাস ও ইন্টারফেস তৈরি করতে হয়।
