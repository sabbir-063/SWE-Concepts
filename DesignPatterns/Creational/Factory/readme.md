# 2. Factory Method Pattern (ফ্যাক্টরি মেথড প্যাটার্ন)

### 📖 মূল কনসেপ্ট (Definition)
**সহজ বাংলায়:** যখন আপনি সরাসরি `new` কিওয়ার্ড ব্যবহার করে অবজেক্ট তৈরি না করে, একটি আলাদা মেথডের (Factory Method) ওপর অবজেক্ট তৈরির দায়িত্ব ছেড়ে দেন, তখন তাকে ফ্যাক্টরি প্যাটার্ন বলে। এটি মূলত অবজেক্ট তৈরির জটিলতা ক্লায়েন্ট থেকে লুকিয়ে রাখে।

**In English:** Factory Method provides an interface for creating objects in a superclass, but allows subclasses to alter the type of objects that will be created.

---

### 💻 কিভাবে কাজ করে? (Implementation Details)
1. একটি কমন ইন্টারফেস বা অ্যাবস্ট্রাক্ট ক্লাস (যেমন: `Vehicle`) থাকে।
2. এর অধীনে কিছু চাইল্ড ক্লাস থাকে (যেমন: `Car`, `Bike`)।
3. একটি ফ্যাক্টরি ক্লাস (যেমন: `VehicleFactory`) থাকে, যার ভেতরে একটি `createVehicle(String type)` মেথড থাকে। 
4. ক্লায়েন্ট ফ্যাক্টরিকে শুধু টাইপের নাম বলে, আর ফ্যাক্টরি সঠিক অবজেক্ট তৈরি করে ফেরত দেয়।

---

### 🧪 বাস্তব জীবনের উদাহরণ (Industry Use Cases)
1. **UI Frameworks:** অ্যান্ড্রয়েড বা ওয়েব ফ্রেমওয়ার্কে বাটন তৈরি করার সময় `createButton("Android")` কল করলে ফ্যাক্টরি অ্যান্ড্রয়েডের বাটন দেয়, আর "iOS" বললে আইওএসের বাটন দেয়।
2. **Logistics System:** ট্রান্সপোর্ট ম্যানেজমেন্টে `createTransport("Road")` কল করলে ট্রাক অবজেক্ট, আর `createTransport("Sea")` কল করলে শিপ (Ship) অবজেক্ট তৈরি হয়।

---

### ✅ সুবিধা (Pros)
- **Open/Closed Principle:** নতুন ধরনের অবজেক্ট (যেমন: `Truck`) যুক্ত করতে ক্লায়েন্টের কোডে হাত দিতে হয় না। শুধু ফ্যাক্টরি ক্লাসে লজিক অ্যাড করলেই হয়।
- ক্লায়েন্টের কোড এবং আসল অবজেক্টের মধ্যে Tight Coupling থাকে না।

### ❌ অসুবিধা (Cons)
- খুব সাধারণ প্রজেক্টের জন্য ফ্যাক্টরি প্যাটার্ন ব্যবহার করলে শুধু শুধু ক্লাস ও ইন্টারফেসের সংখ্যা বাড়ে।
