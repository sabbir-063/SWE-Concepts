# 19. State Pattern (স্টেট প্যাটার্ন)

### 📖 মূল কনসেপ্ট (Definition)
**সহজ বাংলায়:** একটি অবজেক্টের ভেতরের অবস্থা (State) পরিবর্তনের সাথে সাথে যদি তার আচরণও (Behavior) পুরোপুরি বদলে যায়, তখন আমরা স্টেট প্যাটার্ন ব্যবহার করি। এটি দেখতে এমন মনে হয় যেন অবজেক্টটি তার ক্লাসই পরিবর্তন করে ফেলেছে!

ধরুন, আপনার মোবাইল ফোন। যখন আপনার ফোনে কল আসে, ফোনটি কী করবে তা নির্ভর করে ফোনের বর্তমান স্টেট বা অবস্থার ওপর। 
- ফোন যদি "Sound" স্টেটে থাকে, তবে সে জোড়ে রিংটোন বাজাবে।
- যদি "Vibrate" স্টেটে থাকে, তবে সে শুধু কাঁপবে।
- যদি "Silent" স্টেটে থাকে, তবে সে কোনো শব্দ বা কাঁপুনি কিছুই করবে না।
মোবাইল কিন্তু একটাই, শুধু তার ভেতরের স্টেট পরিবর্তনের কারণে তার আচরণ বদলে যাচ্ছে।

**In English:** State pattern lets an object alter its behavior when its internal state changes. It appears as if the object changed its class.

---

### 🧩 প্যাটার্নের কম্পোনেন্টগুলো (Components of the Pattern)
এই প্যাটার্নে ৩টি মূল কম্পোনেন্ট থাকে:

1. **Context (কনটেক্সট):** এটি হলো আমাদের আসল অবজেক্ট (যেমন: `MobileContext`)। এর ভেতরে বর্তমান স্টেটের একটি রেফারেন্স রাখা থাকে।
2. **State (ইন্টারফেস):** এটি সব স্টেটের জন্য কমন মেথডগুলো বলে দেয় (যেমন: `alert()`)।
3. **Concrete States (আসল স্টেট ক্লাস):** এগুলো হলো আলাদা আলাদা অবস্থা (যেমন: `SoundState`, `SilentState`)। প্রতিটা স্টেটের ভেতরে লেখা থাকে, অবজেক্ট এই স্টেটে থাকলে ঠিক কী কাজ করবে।

---

### 💻 কিভাবে কাজ করে? (Java Implementation)
আপনার ফোল্ডারে থাকা `Main.java` ফাইলটি ওপেন করুন। সেখানে একটি মোবাইল ফোনের রিং টোন স্টেটের (Sound, Vibrate, Silent) উদাহরণ দেওয়া আছে।

---

### 🔍 Deep Explanation (কেন বিশাল if-else ব্যবহার করবো না?)
স্টেট প্যাটার্ন না থাকলে আমাদের মোবাইল ক্লাসের কোড দেখতে হতো এমন:
```java
public void alert() {
    if (state == "SOUND") {
        System.out.println("Ring... Ring...");
    } else if (state == "VIBRATE") {
        System.out.println("Bzzzz... Bzzzz...");
    } else if (state == "SILENT") {
        System.out.println("No sound");
    }
}
```
এই বিশাল `if-else` বা `switch` স্টেটমেন্ট মেইনটেইন করা খুব কষ্টের। যদি নতুন কোনো স্টেট "Do Not Disturb (DND)" আসে, তবে আপনাকে এই ক্লাসটি এডিট করতে হবে (Open/Closed Principle ভাঙবে)। 
স্টেট প্যাটার্ন এই বিশাল `if-else` গুলোকে কেটে এক-একটি আলাদা ক্লাসে (State Classes) রূপান্তর করে দেয়। ফলে কনটেক্সট ক্লাসটি একদম ছোট ও পরিষ্কার হয়ে যায়।

### 🧪 বাস্তব জীবনের উদাহরণ (Industry Use Cases)
1. **Media Player:** একটি অডিও প্লেয়ারে 'Play' বাটনের কাজ স্টেট অনুযায়ী বদলায়। প্লেয়ারটি যদি `Playing` স্টেটে থাকে তবে 'Play' বাটন চাপলে সেটা `Pause` হয়ে যায়। আর যদি `Paused` স্টেটে থাকে তবে 'Play' চাপলে গান বাজতে শুরু করে। 
2. **Vending Machine:** একটি ভেন্ডিং মেশিনে টাকা দেওয়ার আগে এক অবস্থা থাকে, টাকা দেওয়ার পর আরেক অবস্থা তৈরি হয় (যেমন: সে তখন প্রডাক্ট সিলেক্ট করতে দেয়)।
3. **TCP Connection:** নেটওয়ার্কিংয়ে টিসিপি কানেকশন বিভিন্ন স্টেটে থাকে (যেমন: ESTABLISHED, LISTENING, CLOSED)। প্রতিটি স্টেটে রিকোয়েস্ট হ্যান্ডেল করার পদ্ধতি আলাদা হয়।

---

### ✅ সুবিধা (Pros)
- বিশাল `if-else` বা `switch` স্টেটমেন্টের হাত থেকে মুক্তি পাওয়া যায়।
- **Single Responsibility Principle:** প্রতিটি স্টেটের নিজস্ব লজিক তার নিজস্ব ক্লাসেই থাকে। 
- নতুন স্টেট তৈরি করা খুব সহজ, কনটেক্সট ক্লাসে হাত দিতে হয় না (Open/Closed Principle)।

### ❌ অসুবিধা (Cons)
- যদি আপনার অবজেক্টের মাত্র দুটি বা তিনটি সাধারণ স্টেট থাকে এবং সেগুলো কখনো পরিবর্তন না হয়, তবে স্টেট প্যাটার্ন ব্যবহার করাটা ওভার-ইঞ্জিনিয়ারিং (Over-engineering) হতে পারে। তখন `if-else` দিয়েই কাজ সারা ভালো।

### 💡 Java Interview Tips
- **Question:** "Strategy Pattern এবং State Pattern-এর মধ্যে পার্থক্য কী? দুটোর স্ট্রাকচার তো প্রায় হুবহু এক!" (এটি ইন্টারভিউয়ের সবচেয়ে কনফিউজিং এবং জনপ্রিয় প্রশ্ন)।
  **Answer:** 
  - **Strategy Pattern-এ** ক্লায়েন্ট নিজে ঠিক করে দেয় কনটেক্সট কোন স্ট্র্যাটেজি ব্যবহার করবে। অর্থাৎ, একবার স্ট্র্যাটেজি সেট করলে সেটা সাধারণত পরিবর্তন হয় না।
  - **State Pattern-এ** কনটেক্সট নিজেই নিজের স্টেট পরিবর্তন করতে পারে। এখানে স্টেটগুলো নিজেদের মধ্যে একে অপরকে চেনে এবং একটি স্টেট থেকে অন্য স্টেটে ট্রান্সফার (Transition) হতে পারে।
