# 18. Observer Pattern (অবজারভার প্যাটার্ন)

### 📖 মূল কনসেপ্ট (Definition)
**সহজ বাংলায়:** অবজারভার প্যাটার্নটি অনেকটা ইউটিউব চ্যানেলের "Subscribe" বাটনের মতো কাজ করে। এটি **Publisher-Subscriber (Pub-Sub)** মেকানিজম ফলো করে। 

ধরুন, আপনি একটি ইউটিউব চ্যানেল সাবস্ক্রাইব করেছেন। চ্যানেলটি যখনই নতুন কোনো ভিডিও আপলোড করবে, সাথে সাথে আপনার কাছে (এবং বাকি সব সাবস্ক্রাইবারদের কাছে) একটি নোটিফিকেশন চলে আসবে। আপনাকে বারবার চ্যানেলে ঢুকে চেক করতে হবে না যে "নতুন ভিডিও এসেছে কি না?"। 

এখানে ইউটিউব চ্যানেলটি হলো **Publisher (Subject)**, আর আপনি হলেন **Subscriber (Observer)**। 

**In English:** Observer pattern defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

---

### 🧩 প্যাটার্নের কম্পোনেন্টগুলো (Components of the Pattern)
এই প্যাটার্নে ৪টি মূল কম্পোনেন্ট থাকে:

1. **Subject / Publisher (ইন্টারফেস):** এর কাজ হলো সাবস্ক্রাইবারদের যুক্ত করা (`subscribe()`), বাদ দেওয়া (`unsubscribe()`) এবং আপডেট পাঠানো (`notify()`)।
2. **Observer / Subscriber (ইন্টারফেস):** এর ভেতরে একটিমাত্র মেথড থাকে, সাধারণত `update()`। পাবলিশার যখনই আপডেট দেয়, সে সবার এই মেথডটা কল করে দেয়।
3. **Concrete Subject (আসল পাবলিশার):** এটি হলো আমাদের আসল ইউটিউব চ্যানেল বা ব্লগ ক্লাস। এটি নিজের কাছে সাবস্ক্রাইবারদের একটি লিস্ট সেভ করে রাখে।
4. **Concrete Observer (আসল সাবস্ক্রাইবার):** এটি হলো আসল ইউজার ক্লাস। এরা আপডেট পেলে কী করবে (স্ক্রিনে দেখাবে নাকি ইমেইল পাঠাবে), সেটা এখানে লেখা থাকে।

---

### 💻 কিভাবে কাজ করে? (Java Implementation)
আপনার ফোল্ডারে থাকা `Main.java` ফাইলটি ওপেন করুন। সেখানে একটি Tech Blog (Publisher) এবং কিছু User (Observer) এর চমৎকার উদাহরণ দেওয়া আছে।

---

### 🔍 Deep Explanation (কেন আমরা Polling করি না?)
অবজারভার প্যাটার্ন না থাকলে কী হতো? সাবস্ক্রাইবারকে প্রতি ৫ মিনিট পরপর একটি লুপ চালিয়ে চেক করতে হতো— `if(blog.hasNewArticle())`। এই বারবার চেক করাকে বলে **Polling (পোলিং)**। পোলিং সিস্টেমের প্রসেসর এবং মেমরি অনেক নষ্ট করে।
অবজারভার প্যাটার্ন পোলিংয়ের সমস্যাটি দূর করে একে **Push Mechanism**-এ রূপান্তর করে। অর্থাৎ ইউজার চেক করবে না, পাবলিশার নিজে থেকেই ইউজারের কাছে ডাটা পুশ করে দিবে।

---

### 🧪 বাস্তব জীবনের উদাহরণ (Industry Use Cases)
1. **DOM Event Listeners:** জাভাস্ক্রিপ্টে আমরা যখন বাটনে `button.addEventListener('click', function)` লিখি, সেটা আসলে অবজারভার প্যাটার্ন! এখানে বাটনটি হলো Publisher, আর আমাদের ফাংশনটি হলো Subscriber।
2. **React/Redux/Vue:** ফ্রন্টএন্ড ফ্রেমওয়ার্কগুলোতে স্টেট (State) চেঞ্জ হলে যে অটোম্যাটিক্যালি UI আপডেট হয়ে যায়, তার পেছনে এই অবজারভার প্যাটার্ন কাজ করে। UI কম্পোনেন্টগুলো স্টেটের সাবস্ক্রাইবার হিসেবে থাকে।
3. **MVC Architecture:** Model-View-Controller আর্কিটেকচারে, Model আপডেট হওয়ার সাথে সাথে View অটোম্যাটিক আপডেট হয়— এটিও অবজারভার প্যাটার্ন দিয়ে করা হয়।
4. **RxJava / Reactive Programming:** পুরো Reactive Programming কনসেপ্টটিই অবজারভার প্যাটার্নের ওপর দাঁড়িয়ে আছে (Observables and Observers)।

---

### ✅ সুবিধা (Pros)
- **Open/Closed Principle:** পাবলিশারের কোড পরিবর্তন না করেই আপনি চাইলে নতুন হাজারটা সাবস্ক্রাইবার ক্লাস যুক্ত করতে পারবেন।
- সাবজেক্ট এবং অবজারভারের মাঝে সম্পর্কটি খুবই ঢিলেঢালা (Loosely coupled)। তারা একে অপরের ডিটেইলস জানে না।

### ❌ অসুবিধা (Cons)
- সাবস্ক্রাইবারদের কাছে নোটিফিকেশনগুলো র্যান্ডম অর্ডারে (Random order) যেতে পারে। অর্থাৎ কে আগে আপডেট পাবে আর কে পরে, তার গ্যারান্টি দেওয়া কঠিন।
- যদি সাবস্ক্রাইবার মেমরি থেকে ঠিকমতো Unsubscribe না হয়, তবে মেমরি লিক (Memory leak) হতে পারে (যাকে 'Lapsed listener problem' বলা হয়)।

### 💡 Java Interview Tips
- **Question:** "Observer Pattern এবং Mediator Pattern-এর মধ্যে পার্থক্য কী?"
  **Answer:** 
  - **Observer:** এটি One-to-Many রিলেশনশিপ। একজন পাবলিশ করে, অনেকেই শুধু শোনে (One way communication)।
  - **Mediator:** এটি Many-to-Many রিলেশনশিপকে কন্ট্রোল করে। সবাই সবার সাথে কথা বলে, কিন্তু মাঝখানে মিডিয়েটর হয়ে (Two way communication)।
