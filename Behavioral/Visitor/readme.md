# 22. Visitor Pattern (ভিজিটর প্যাটার্ন)

### 📖 মূল কনসেপ্ট (Definition)
**সহজ বাংলায়:** আপনার কাছে যদি একটি ডাটা স্ট্রাকচার থাকে (যেমন: অনেকগুলো আলাদা আলাদা ক্লাসের অবজেক্টের একটি লিস্ট), এবং আপনি সেই সবগুলো অবজেক্টের ওপর একটি **নতুন অপারেশন বা কাজ (Operation)** চালাতে চান, কিন্তু **আসল অবজেক্টগুলোর কোড পরিবর্তন করতে চান না**— তখন ভিজিটর প্যাটার্ন ব্যবহার করা হয়।

ধরুন, আপনার একটি শপিং কার্ট আছে। সেখানে `বই`, `ফল`, `জামা`— এমন নানান টাইপের ক্লাস (অবজেক্ট) রাখা আছে। এখন আপনাকে বলা হলো এই সবকিছুর ডিসকাউন্ট হিসাব করতে হবে। 
সাধারণত আপনি কী করতেন? আপনি `বই`, `ফল` প্রতিটি ক্লাসের ভেতর গিয়ে একটি `calculateDiscount()` নামের মেথড লিখতেন। কিন্তু এতে মূল ক্লাসগুলো বারবার এডিট করতে হয় (Open/Closed Principle ভাঙে)। 
এর বদলে আপনি একটি "Visitor" (ভিজিটর/গেস্ট) ক্লাস বানাবেন। সেই ভিজিটরটি `বই`-এর কাছে যাবে, তার ডিসকাউন্ট হিসাব করবে, এরপর `ফল`-এর কাছে যাবে, তার ডিসকাউন্ট হিসাব করবে। ভিজিটরের কাছেই সব লজিক থাকবে, আসল ক্লাসগুলো (বই, ফল) একদম আগের মতোই থাকবে!

**In English:** Visitor pattern separates an algorithm from the objects on which it operates. It lets you add new operations to existing object structures without modifying those structures.

---

### 🧩 প্যাটার্নের কম্পোনেন্টগুলো (Components of the Pattern)
এই প্যাটার্নে ৪টি মূল কম্পোনেন্ট থাকে:

1. **Visitor Interface (ভিজিটর ইন্টারফেস):** এটি রুলস ঠিক করে দেয় যে, ভিজিটর কোন কোন ক্লাসে ভিজিট করতে পারবে। (এর ভেতরে `visit(Book)`, `visit(Fruit)` এরকম মেথড থাকে)।
2. **Concrete Visitor (আসল ভিজিটর):** এটি হলো সেই ক্লাস যার ভেতরে লজিক বা অ্যালগরিদমটি লেখা থাকে (যেমন: `PriceCalculatorVisitor` বা `TaxCalculatorVisitor`)।
3. **Element Interface (এলিমেন্ট ইন্টারফেস):** যাকে ভিজিট করা হবে, তার ইন্টারফেস। এর ভেতর একটিমাত্র মেথড থাকে— `accept(Visitor v)`। এর মাধ্যমে সে ভিজিটরকে তার ভেতরে ঢোকার অনুমতি দেয়।
4. **Concrete Elements (আসল অবজেক্টগুলো):** এগুলো হলো আমাদের ডাটা স্ট্রাকচারের আইটেম (যেমন: `Book`, `Fruit`)।

---

### 💻 কিভাবে কাজ করে? (Java Implementation)
আপনার ফোল্ডারে থাকা `Main.java` ফাইলটি ওপেন করুন। সেখানে একটি শপিং কার্টের উদাহরণ দেওয়া আছে, যেখানে `PriceCalculatorVisitor` শপিং কার্টের প্রতিটি আইটেম ভিজিট করে তার দাম এবং ডিসকাউন্ট হিসাব করছে।

---

### 🔍 Deep Explanation (Double Dispatch - ডাবল ডেসপ্যাচ)
ভিজিটর প্যাটার্নের সবচেয়ে ট্রিকি এবং শক্তিশালী কনসেপ্ট হলো **"Double Dispatch"**। 
জাভাতে আমরা যখন একটি মেথড কল করি (Single Dispatch), তখন কোন ক্লাসের মেথড কল হবে তা রান-টাইমে ঠিক হয়। কিন্তু ভিজিটর প্যাটার্নে মেথড কল দুটি জিনিসের ওপর নির্ভর করে:
১. কোন ভিজিটর কল করছে? (যেমন: `TaxVisitor` নাকি `DiscountVisitor`)
২. কোন এলিমেন্ট কল রিসিভ করছে? (যেমন: `Book` নাকি `Fruit`)

যখন আপনি `item.accept(visitor)` কল করেন, তখন প্রথমে এলিমেন্টের ধরন অনুযায়ী `accept` মেথডটি ট্রিগার হয়। এরপর এলিমেন্টটি আবার `visitor.visit(this)` কল করে ভিজিটরের কাছে যায়। এই দুই ধাপে মেথড কলিংকেই বলা হয় Double Dispatch।

### 🧪 বাস্তব জীবনের উদাহরণ (Industry Use Cases)
1. **Compilers / Syntax Trees:** ভিজিটর প্যাটার্নের সবচেয়ে বড় রিয়েল-লাইফ ব্যবহার হলো কম্পাইলার (Compiler) ডিজাইনে! একটি Abstract Syntax Tree (AST)-তে অনেক ধরনের নোড (Node) থাকে। এই নোডগুলোকে কোড জেনারেট করা, অপটিমাইজ করা বা ভুল ধরার জন্য বারবার পরিবর্তন না করে, বিভিন্ন Visitor (যেমন: `TypeCheckVisitor`, `CodeGenerateVisitor`) বানিয়ে নোডগুলোর ওপর পাঠানো হয়।
2. **Exporting Data:** আপনার কাছে অনেকগুলো ক্লাসের অবজেক্টের একটি লিস্ট আছে। এখন আপনার বস বললো এগুলোকে JSON ফরম্যাটে এক্সপোর্ট করতে হবে। আপনি ক্লাসগুলোর ভেতর `exportToJson()` না লিখে, একটি `JsonExportVisitor` বানাতে পারেন, যে সবার কাছে গিয়ে ডাটা নিয়ে JSON বানিয়ে দিবে। কাল যদি XML এক্সপোর্ট করতে হয়, তবে শুধু আরেকটি `XmlExportVisitor` বানালেই হবে!

### ✅ সুবিধা (Pros)
- **Open/Closed Principle:** আপনি existing ক্লাসগুলোর (বই, ফল) কোনো পরিবর্তন না করেই তাদের ওপর হাজারটা নতুন অপারেশন (Tax, Discount, Export) যুক্ত করতে পারবেন।
- **Single Responsibility Principle:** একটি নির্দিষ্ট কাজ (যেমন: দাম হিসাব করা) একটি নির্দিষ্ট ভিজিটর ক্লাসেই থাকে।

### ❌ অসুবিধা (Cons)
- যদি আপনার অবজেক্ট স্ট্রাকচারে নতুন কোনো ক্লাস (যেমন: `Electronics`) যুক্ত হয়, তবে আপনাকে `Visitor` ইন্টারফেস এবং তার সমস্ত Concrete Visitor ক্লাসগুলোতে গিয়ে `visit(Electronics)` মেথডটি ম্যানুয়ালি লিখতে হবে!
- ভিজিটরকে অনেক সময় এলিমেন্টের প্রাইভেট ফিল্ডগুলোর অ্যাক্সেস দেওয়া লাগে, যা এনক্যাপসুলেশন (Encapsulation) নষ্ট করতে পারে।

### 💡 Java Interview Tips
- **Question:** "কখন ভিজিটর প্যাটার্ন ব্যবহার করা উচিত নয়?"
  **Answer:** যখন আপনার ডাটা স্ট্রাকচারের ক্লাসগুলো (Elements) ঘন ঘন পরিবর্তন হয় বা নতুন ক্লাস যুক্ত হয়। কারণ একটি নতুন এলিমেন্ট যুক্ত হওয়া মানেই সব ভিজিটর ক্লাসে গিয়ে কোড চেঞ্জ করা। ভিজিটর তখনই ভালো যখন এলিমেন্ট ক্লাসগুলো ফিক্সড বা স্টেবল (Stable) থাকে, কিন্তু অপারেশন বা লজিক বারবার যুক্ত হয়।
